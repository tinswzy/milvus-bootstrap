# 方案二：用 K8s 原生操作拆掉其中一个服务（不改 helm/operator）

> 面向读者：研发 / SRE。
>
> 目标：在**完全不改** milvus-helm / milvus-operator 代码的前提下，用 `kubectl` / `helm` 原生操作，从一个已安装的 Milvus 部署里**安全地拆掉其中一个中间件服务**（以删除 **Kafka**、切到 **Woodpecker** 为例），并完成 `mq.type` 切换与 Milvus 滚动重建。
>
> 定位：**当下就能用**的止血手册，零代码风险。代价是命令多、顺序敏感、无回滚护栏——体验差，详见文末"为什么体验差"。

---

## 0. 重要前提（务必先读）

> 这几条不读，后面每一步都可能造成**数据丢失**。

1. **MQ 不可热切、数据不可迁移**。Milvus 内核里 `mq.type` 实质不可变（`milvus/internal/util/streamingutil/util/wal_selector.go:46-74`），不同 MQ 后端的 WAL 格式互不相通。**所以"切 MQ" = 在一个干净/空的集群上换后端，而不是带着存量数据原地迁移。**
2. **执行前必须确认集群可清空**：要么是新实例，要么已对存量 collection 做了导出/备份（用 [milvus-backup](https://github.com/zilliztech/milvus-backup) 或 Birdwatcher 导出），且能接受 drop 后重建并重新导入。
3. **删除 PVC 不可逆**。本手册默认**保留** PVC，删 PVC 是单独的、显式的最后一步。
4. **区分安装方式**：你的 Milvus 是 **helm 装的**还是 **operator 装的**？两者拆服务的方式不同，下面分别给出。

---

## 1. 先盘清现状

```bash
NS=milvus                      # 改成你的 namespace
REL=my-release                 # helm release 名 / operator 的 Milvus 名

# 1.1 看 Milvus 当前连的是哪个 MQ
kubectl -n $NS get cm ${REL}-milvus -o jsonpath='{.data.*}' | grep -A2 -E '^mq:|messageQueue'

# 1.2 列出所有 workload,认出 kafka / pulsar / etcd / minio / woodpecker
kubectl -n $NS get sts,deploy,svc,pvc -l app.kubernetes.io/instance=$REL

# 1.3 如果是 helm 装的,看依赖子 release / 子 chart
helm -n $NS get values $REL | grep -E 'enabled|messageQueue' 
helm -n $NS list
```

判断安装方式：

```bash
# 有这个 CR → operator 装的
kubectl -n $NS get milvus $REL 2>/dev/null && echo "==> operator-managed" || echo "==> helm-managed"
```

---

## 2. 路线 A：operator 装的，拆掉 Kafka 切 Woodpecker

operator 会持续 reconcile，所以**不能直接 kubectl delete kafka 资源**（会被拉起来）。要改 CR 让 operator 自己收手。

### A-1. 确认集群已清空 / 已备份（见前提）

### A-2. 改 Milvus CR：把 active MQ 切到 woodpecker

```bash
kubectl -n $NS edit milvus $REL
```

```yaml
spec:
  mode: cluster
  dependencies:
    msgStreamType: woodpecker          # ← 从 kafka 改成 woodpecker
    woodpecker:
      persistence:
        enabled: true
    # woodpecker 依赖对象存储,确认 storage 已配置
    storage:
      external: false                  # 用内置 minio,或 external:true 指向 S3
    kafka:
      inCluster:
        deletionPolicy: Delete         # ← 允许 operator 卸载 kafka release
        pvcDeletion: false             # ← 先保留 PVC,确认无误再单独删
```

> `setDefaultMsgStreamConfigs`（`milvus-operator/apis/.../milvus_webhook.go:498-537`）会为 woodpecker 补默认配置。`ReconcileMsgStream`（`pkg/controllers/dependencies.go`）发现 type 不再是 kafka，就**不再 reconcile kafka**。

### A-3. operator 触发清理

operator 的 finalizer/清理逻辑（`pkg/controllers/milvus.go:103-130`）会在 `DeletionPolicy=Delete` 时把 `${REL}-kafka` 这个 helm release 卸载掉。观察：

```bash
kubectl -n $NS get pods -w           # 看 kafka pod 逐步消失, woodpecker / milvus 重建
kubectl -n $NS get milvus $REL -o jsonpath='{.status.status}{"\n"}'
```

> 若 operator 未自动卸载 kafka 子 release（取决于版本），手动补一刀：
> ```bash
> helm -n $NS uninstall ${REL}-kafka || kubectl -n $NS delete sts,svc,cm -l app.kubernetes.io/instance=${REL}-kafka
> ```

### A-4. 校验

```bash
kubectl -n $NS get cm ${REL}-milvus -o yaml | grep -A1 '^  type:'   # 应为 woodpecker
kubectl -n $NS get pods | grep -E 'kafka|zookeeper'                 # 应为空
```

## 3. 路线 B：helm 装的，拆掉 Kafka 切 Woodpecker

helm 没有 operator 那种持续 reconcile，所以是"改 values → `helm upgrade` → 收尾"。

### B-1. 关掉 kafka、打开 woodpecker，切 mq.type

把现有 values 导出再改：

```bash
helm -n $NS get values $REL > values.now.yaml
```

编辑 `values.now.yaml`：

```yaml
cluster:
  enabled: true

# 关掉 kafka 子 chart(condition:kafka.enabled → false ⇒ helm 删除其资源)
kafka:
  enabled: false

# 打开 woodpecker(它是手写 STS,不是子 chart)
woodpecker:
  enabled: true

# 确保对象存储在位(woodpecker 依赖)
minio:
  enabled: true            # 或用 externalS3
```

应用：

```bash
helm -n $NS upgrade $REL milvus/milvus -f values.now.yaml
```

> `config.tpl` 的 if-else 链此时会渲染出 `mq.type: woodpecker`（kafka 分支已关）。`helm upgrade` 会删除 kafka 子 chart 的 Deployment/STS/SVC，**但不会删 kafka 的 PVC**。

### B-2. 滚动重建 Milvus 组件，让新 ConfigMap 生效

ConfigMap 变了，但已运行的 pod 不会自动重读。强制重启：

```bash
kubectl -n $NS rollout restart deploy -l app.kubernetes.io/instance=$REL
kubectl -n $NS rollout restart sts    -l app.kubernetes.io/instance=$REL
kubectl -n $NS rollout status  deploy -l app.kubernetes.io/instance=$REL,component=proxy
```

### B-3. 验证 mq.type 已切换

```bash
kubectl -n $NS exec deploy/${REL}-milvus-proxy -- \
  sh -c 'grep -A1 "^mq:" /milvus/configs/default.yaml'   # type: woodpecker
```

---

## 4. 收尾：清理残留的 Kafka PVC（不可逆，最后做）

确认 woodpecker 跑稳、业务验证通过后，再删 kafka 遗留的 PVC（否则白占存储）：

```bash
# 先看清楚要删哪些
kubectl -n $NS get pvc | grep -E 'kafka|zookeeper'

# 确认无误后删除(不可逆!)
kubectl -n $NS delete pvc -l app.kubernetes.io/instance=${REL},app.kubernetes.io/name=kafka
kubectl -n $NS delete pvc -l app.kubernetes.io/instance=${REL},app.kubernetes.io/name=zookeeper
```

---

## 5. 回滚预案

| 阶段 | 出问题怎么回滚 |
| --- | --- |
| 改了 CR / values 但还没删 PVC | operator：把 `msgStreamType` 改回 `kafka`；helm：`helm rollback $REL <上个 revision>`。kafka PVC 还在，数据可恢复 |
| 已删 kafka workload，PVC 还在 | 重新 `kafka.enabled: true` / 改回 CR，PVC 复用，kafka 起回来 |
| **已删 PVC** | **无法回滚**，只能从备份重建 + 重新导入数据 |

> 所以**第 4 步（删 PVC）务必放到最后、且二次确认**。

## 6. 操作检查清单（打印出来逐项打勾）

- [ ] 已确认集群可清空或已完成数据备份/导出
- [ ] 已记录当前 `mq.type` 与 helm revision（`helm history $REL`）
- [ ] 已识别安装方式（operator / helm）
- [ ] （operator）已改 CR `msgStreamType` 并设 `deletionPolicy/pvcDeletion`
- [ ] （helm）已改 values 关 kafka / 开 woodpecker 并 `helm upgrade`
- [ ] 已 `rollout restart` 并确认新 ConfigMap 生效
- [ ] 已验证 kafka/zookeeper pod 消失、`mq.type=woodpecker`
- [ ] 业务验证通过后，才删除 kafka PVC（不可逆）

---

## 7. 为什么"体验很差"

| 痛点 | 具体表现 |
| --- | --- |
| **命令多、顺序敏感** | 改 CR/values → upgrade → restart → 校验 → 删 PVC，漏一步或顺序错就出问题 |
| **无护栏** | 没有任何东西阻止你在有数据时直接切 MQ、或提前删 PVC，全靠人记得前提 |
| **operator/helm 两套手法** | 同一目标，两种完全不同的命令路径，运维要分别掌握 |
| **状态不可视** | "kafka 到底卸干净没""mq.type 切了没"要靠一条条 grep 去确认，没有统一视图 |
| **无回滚一键** | 删 PVC 后不可逆，回滚要靠人脑记住分支 |
| **易错点都藏在前提里** | 数据不可迁移、PVC 不可逆这些最致命的点，文档写了但执行时无人提醒 |

**结论**：方案二零代码、当天可用，是 3.0 上线时最现实的兜底。但它把所有复杂度和风险都压在**人**身上。这正是 [方案三](03-方案三_enterprise-admin-webui.md) 要解决的：把这套手册里的每一步、每一个前提校验、每一次二次确认，都变成 admin webui 里一个**有护栏的向导**。
