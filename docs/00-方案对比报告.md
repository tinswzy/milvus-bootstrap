# Milvus 3.0 资源解耦方案对比报告

> 主题：为满足 Milvus 3.0 **switch MQ** 功能，如何解耦"底层各资源的安装"与"Milvus 的实际运行连线"。
>
> 面向读者：架构 / 技术管理决策层。
>
> 配套工程文档：[方案一](01-方案一_helm-operator扩展.md)、[方案二](02-方案二_k8s原生操作.md)、[方案三](03-方案三_enterprise-admin-webui.md)。

---

## 1. 背景与问题本质

### 1.1 Milvus 3.0 在内核层已经做对了一半

Milvus 3.0 引入了流式（Streaming）架构：`MixCoord` 内置 `StreamingCoord`，由 `StreamingNode` 管理一批 PChannel，所有 MQ 后端都被收敛到统一的 **WAL 接口**之后：

```go
// milvus/pkg/streaming/walimpls/wal.go
type WALImpls interface {
    ROWALImpls
    Append(ctx context.Context, msg message.MutableMessage) (message.MessageID, error)
    Truncate(ctx context.Context, id message.MessageID) error
}
```

也就是说，从**内核视角**看，`pulsar / kafka / woodpecker / rocksmq` 已经是"可插拔的 WAL 实现"，由 `mq.type` 一个开关决定：

```yaml
# milvus/configs/milvus.yaml:167-186
# Milvus supports four message queues: rocksmq, Pulsar, Kafka, and Woodpecker.
# You can change the MQ by setting the mq.type field.
mq:
  type: default     # [default, pulsar, kafka, rocksmq, woodpecker]
```

并且 3.0 推出了 Milvus 原生的 **Woodpecker**（基于对象存储的内置 WAL/日志存储，`configs/milvus.yaml:188-228`），官方建议新实例直接用 Woodpecker，以获得"更好的性能、更简单的运维和更低的成本"。

> **结论 A**：换 MQ 在**内核层面**只是改一个 `mq.type` + 提供对应连接信息。内核已经准备好了。

### 1.2 真正的耦合在"安装编排"这一层

问题出在部署工具。`milvus-helm` 和 `milvus-operator` 把两件**本应独立**的事，搅成了一件事：

| 本应独立的两件事 | 现状（被耦合） |
| --- | --- |
| **A. 安装编排**：谁负责把 etcd / minio / pulsar / kafka 这些**独立中间件**装到 K8s 上、谁负责它们的生命周期 | 由 milvus-helm 的子 chart / milvus-operator 的 reconcile 越界代管 |
| **B. 运行时连线**：milvus 进程启动时该连哪个 etcd、哪个 MQ、用什么地址 | 由同一套工具在 install/upgrade 时**烘焙**进渲染后的 ConfigMap |

这两件事被**同一套 if-else 互斥假设**绑死了。证据：

**Helm 侧**——`charts/milvus/templates/config.tpl` 用一条**带优先级的 if-else-if 链**决定 MQ 并把地址注入渲染后的 `milvus.yaml`：

```
IF  externalPulsar.enabled  → type: pulsar (外部地址)
ELSE IF pulsar.enabled       → type: pulsar (内置 v2)
ELSE IF woodpecker.enabled   → type: woodpecker
ELSE IF pulsarv3.enabled     → type: pulsar (内置 v3, 默认开)
（再独立判断）
IF  externalKafka.enabled    → type: kafka (外部)
ELSE IF kafka.enabled        → type: kafka (内置)
```

同时这些 MQ 是 chart 的**条件依赖子 chart**（`charts/milvus/requirements.yaml`：etcd / minio / pulsarv2 / pulsarv3 / kafka，按 `*.enabled` 开关），Woodpecker 甚至是手写的 StatefulSet。**"装不装 pulsar" 和 "milvus 连不连 pulsar" 共用了同一个 `pulsar.enabled` 开关。**

**Operator 侧**——`MilvusDependencies` 把所有依赖塞进一个结构体，`MsgStreamType` 是一个**互斥枚举**：

```go
// milvus-operator/apis/milvus.io/v1beta1/dependencies_types.go
const (
    MsgStreamTypePulsar     MsgStreamType = "pulsar"
    MsgStreamTypeKafka      MsgStreamType = "kafka"
    MsgStreamTypeWoodPecker MsgStreamType = "woodpecker"
    MsgStreamTypeRocksMQ    MsgStreamType = "rocksmq"
    MsgStreamTypeNatsMQ     MsgStreamType = "natsmq"
    MsgStreamTypeCustom     MsgStreamType = "custom"
)
```

`pkg/controllers/dependencies.go` 的 `ReconcileMsgStream` 按这个枚举 `switch`，决定**装哪个 MQ**：

```go
func (r *MilvusReconciler) ReconcileMsgStream(ctx, mc) error {
    switch mc.Spec.Dep.MsgStreamType {
    case MsgStreamTypeKafka:  return r.ReconcileKafka(ctx, mc)   // helm 装 kafka
    case MsgStreamTypePulsar: return r.ReconcilePulsar(ctx, mc)  // helm 装 pulsar
    default:                  return nil                          // 内置 MQ 啥也不装
    }
}
```

**一个 `MsgStreamType` 枚举，同时决定了"装哪个中间件"和"milvus 连哪个 MQ"。** 这就是耦合点。

### 1.3 为什么"switch MQ"会被这个耦合卡住

因为这套工具隐含了一个假设：**"一个 Milvus 实例只会有一种 MQ，且这种 MQ 由我（helm/operator）负责装"**。于是：

1. **无法并存 / 灰度**：要从 woodpecker 切到 kafka，互斥的 if-else 不允许两者同时存在，无法"先装好 kafka、校验通过、再切流"。
2. **删除半条命**：切换后，旧 MQ（如 pulsar 子 chart）的卸载依赖 operator finalizer 的 `DeletionPolicy`（`pkg/controllers/milvus.go:103-130`），用户对"到底删没删、PVC 清没清"几乎没有可视的、独立的控制。
3. **越界代管**：kafka / pulsar / etcd / minio 本来都有**自己官方的 operator 和 CRD**，却被 milvus-helm/operator 用一套简化子 chart 代管，能力受限、版本滞后、和官方生态割裂。

> **结论 B**：switch MQ 的障碍不在内核，而在"安装编排"和"运行连线"被同一套互斥 if-else 绑死。**解法的核心，是把这两件事拆开。** 三个方案的区别，只是"拆到什么程度、由谁来拆"。

### 1.4 一个绕不开的硬约束（三方案共同的前提）

无论哪个方案，都要正视一条内核约束：

- **`mq.type` 实质不可热切换**，且**WAL 数据在不同 MQ 后端之间不可迁移**（格式各异）。`wal_selector.go:46-74` 在选定后即固定；woodpecker 在 cluster 模式还禁止 local storage。

这意味着**"switch MQ" ≈ "在干净/空集群上换后端"或"新建实例时选后端"**，而**不是**"带着存量数据原地热迁移"。任何方案都必须把这条约束**显式告知用户**，而不能假装可以无损切换。三方案对这条约束的"体验"差别很大（见对比表"风险可视化"一行）。

## 2. 三个方案速览

| | 方案一：扩展 helm/operator | 方案二：K8s 原生操作 | 方案三：Enterprise Admin WebUI |
| --- | --- | --- | --- |
| 一句话 | 继续在现有 helm/operator 里改 if-else，让它支持多 MQ 并存与删除单个 MQ | 不改工具，安装后由人用 kubectl/helm 手工拆掉其中一个服务 | 做一个轻量 Python+Web 控制台，编排各组件**自己的**原生 operator/helm |
| 解耦程度 | 低（still 一套工具代管一切） | 中（运行时由人手工解耦，但工具仍耦合） | 高（安装与运行彻底分离，组件各管各的） |
| 改内核/工具量 | 中（改 chart 模板 + operator reconcile/webhook） | 小（基本不改代码，写手册） | 大（新建一个产品） |
| 用户体验 | 中（YAML 一把梭，但语义越来越绕） | 差（一堆命令、易错、无回滚护栏） | 好（所见即所得，搭积木） |
| 纯粹度 | ↓ 越改越不纯粹 | — 工具不变，但靠人肉补 | ↑ 最纯粹，组件各归各 |
| 上线周期 | 数周 | 数天（文档为主） | 数月（产品化） |

## 3. 多维对比表

> 评分：●●●（优） / ●●○（中） / ●○○（弱）。"纯粹度"指是否避免了"一个工具越界代管别人"。

| 维度 | 方案一 扩展 helm/operator | 方案二 K8s 原生操作 | 方案三 Admin WebUI |
| --- | --- | --- | --- |
| **实现成本** | ●●○ 改模板/reconcile/webhook，有回归面 | ●●● 几乎不写代码，产出是文档 | ●○○ 从零做一个 Python+Web 产品 |
| **上线周期** | 数周 | 数天 | 数月（可分期） |
| **架构纯粹度/解耦** | ●○○ 仍是"一套工具管一切"，且越改越绕 | ●●○ 运行期靠人解耦，工具本身仍耦合 | ●●● 安装与运行彻底分离，组件各用自己 operator |
| **用户体验** | ●●○ 还是改 values/CR，但多 MQ 语义反直觉 | ●○○ 命令多、顺序敏感、无护栏、易翻车 | ●●● 向导式、所见即所得、有校验与回滚提示 |
| **对其他组件的侵入** | ●○○ 继续越界代管 kafka/pulsar/etcd/minio | ●●○ 拆的时候直面原生资源，但仍是 helm 装的简化版 | ●●● 直接用 kafka/etcd/minio 官方 operator，不越界 |
| **风险可视化**（mq 不可迁移、删 PVC） | ●○○ 藏在 finalizer/DeletionPolicy 里，用户难感知 | ●○○ 全靠人记得，漏删/误删风险高 | ●●● 向导显式弹"数据不可迁移""将删除 PVC"确认 |
| **多 MQ 并存 / 灰度切换** | ●●○ 改造后可支持，但模板复杂度飙升 | ●○○ 理论可行，全手工编排，几乎不可控 | ●●● 天然支持"先装新 MQ→校验→切流→下线旧 MQ" |
| **可维护性 / 回归面** | ●○○ if-else 组合爆炸，测试矩阵变大 | ●●○ 工具不变，维护的是文档 | ●●● 编排逻辑集中在一处，组件升级跟随各自 operator |
| **与上游生态对齐** | ●○○ 自己维护 pulsar/kafka 子 chart，滞后 | ●●○ | ●●● 跟随 Strimzi/Pulsar/etcd 官方 operator |
| **对存量用户兼容** | ●●● 无缝，老 values/CR 继续用 | ●●● 不改工具，老部署不受影响 | ●●○ 新增管理面，老部署可逐步纳管 |
| **审计 / 多人协作 / 权限** | ●○○ 取决于 GitOps 自建 | ●○○ 命令行，难审计 | ●●● 控制台天然可加操作日志/RBAC |

## 4. 推荐路线

三个方案**不是单选**，而是一条时间线：

```
  现在 ──────────────► 3.0 GA ──────────────► 3.x 战略
  方案二（手册止血）     方案一（工具支持）      方案三（产品解耦）
  几天可交付            数周内补齐             数月分期建设

  立刻能用              覆盖大多数 switch       彻底解决"纯粹度"
  但体验差              MQ 场景               与"组件各管各"
```

**推荐组合：**

1. **T0（随 3.0）— 方案二打底**：先把"安装后如何用 kubectl/helm 安全地拆掉一个 MQ、改 `mq.type`、滚动重建"写成**可执行手册**（[详见方案二](02-方案二_k8s原生操作.md)），让 switch MQ 当下就有路可走。零代码风险，几天交付。
2. **T1（3.0.x）— 方案一补工具**：在 helm/operator 里**有限度**地放开 MQ 互斥假设、让 `upgrade` 支持安全删除单个 MQ 服务（[详见方案一](01-方案一_helm-operator扩展.md)）。把方案二的手册步骤"工具化"，降低翻车率。**但明确这是过渡——每加一种组合，模板就更不纯粹一分。**
3. **T2（3.x）— 方案三定方向**：启动 **Milvus Enterprise Admin WebUI**（[详见方案三](03-方案三_enterprise-admin-webui.md)）。把"安装编排"从 milvus-helm/operator 中彻底剥离，每个组件交还给它**自己的**官方 operator，admin webui 只做三件事：**(a) 搭积木式编排安装、(b) 改各组件 ConfigMap 并触发重启、(c) 把组件间的依赖规则在界面层显式表达**。这才是"纯净丝滑"的终态。

**一句话给决策层**：方案二是创可贴，方案一是夹板，方案三是把骨头接正。前两者解决"能不能切"，第三者解决"为什么切这么难"。

## 5. 风险与开放问题

| 风险 / 问题 | 说明 | 缓解 |
| --- | --- | --- |
| **MQ 数据不可迁移** | `mq.type` 实质不可热切，WAL 跨后端不可迁移；switch ≈ 空集群换后端 | 三方案都必须显式提示；方案三在向导里强制确认 |
| **删除 PVC 数据丢失** | 旧 MQ 下线时 PVC 清理（`DeletionPolicy=Delete` + `PVCDeletion`）不可逆 | 默认 `Retain`，删除前二次确认 + 备份提示 |
| **方案一回归面** | if-else 组合增多，标准/集群 × pulsar/kafka/woodpecker/外部 矩阵膨胀 | 优先做"删除单 MQ"，并存/灰度作为可选项分期 |
| **方案三人力** | 等于立项一个新产品（前端 + Python 后台 + 各 operator 集成） | 先交付只读纳管 + 核心 switch MQ 向导，再扩组件 |
| **谁来装官方 operator** | 方案三依赖 Strimzi(kafka)/Pulsar/etcd/MinIO operator 已就位 | admin webui 提供 operator 自身的一键安装/检测 |
| **on-prem 环境差异** | 客户集群 K8s 版本、存储类、网络策略各异 | 方案三做环境预检（preflight）页 |
