方案对比 · 为什么 switch MQ 这么难

为满足 Milvus 3.0 的 switch MQ，需要解耦"底层资源的安装"与"Milvus 的运行连线"。三种方案分别从改工具 / 手工操作 / 做产品三个方向解。

一句话：「装哪个中间件」（provision）与「milvus 连哪个 MQ」（active）本是两件事，却被 helm/operator 用同一个开关绑死。三方案的区别，只是在哪一层、由谁来把它拆开。

问题本质 · 一个开关，被迫干两件事

Milvus 3.0 内核已把 MQ 收敛到统一的 WALImpls 接口之后，「连哪个 MQ」本该只是一个 mq.type 开关 —— 内核早就准备好了。卡点在部署工具：它用同一个开关，同时决定了「装哪个中间件」和「milvus 连哪个 MQ」。

被同一开关绑死
⇒ 表达不出「kafka 已装好 · milvus 还连着 woodpecker」这种切换中间态，也就没法「先装新 MQ → 校验 → 切流 → 下线旧 MQ」。
现状代码：helm 是一条 if externalPulsar → … else if woodpecker → … else if pulsarv3 优先级链；operator 是 switch MsgStreamType 互斥枚举。其中 kafka 在 helm 里还是另起的独立 if 块，会盖掉前面的链 —— 这正是"越改越绕"的活样本。
解法 · 装连分离（三方案共同的内核）
装 provision = { woodpecker, kafka } 集合 · 可多选（切换窗口里并存）
连 active = woodpecker → kafka 单值 · 单选（同一刻只连一个）
⇒ 先装新 MQ、校验通过再切流、最后下线旧 MQ —— 每步可控、可回滚。三方案的区别，只是在哪一层、由谁来做这件拆分。
硬约束（三方案都绕不开）：不同 MQ 后端的 WAL / checkpoint 格式互不相通，mq.type 选定后即固定，跨后端不可无损迁移 —— 所以 switch MQ ≈ 先备份 / 清空，再换后端，不是带存量数据热迁移。任何方案都必须把这条显式告知用户。

理想终态 · k8s 原生方式，各组件各用官方 operator

三方案基本思路

同一个"装/连分离"，三种力度
方案一 · 过渡 趁 3.0 改造 helm/operator，把安装与配置解耦

保持现在 helm/operator "一把装齐所有依赖" 的形态，但做一处关键改造：把【安装】和【配置】彻底拆开。安装工具不再假设依赖之间互斥，只负责一波 install / uninstall / update；至于 Milvus 连哪个 MQ、连哪个存储，是 configmap.*.* 模板渲染的事，与"装了哪些组件"无关。两个问题互相独立。这种结构性改动趁 3.0 一起改，越往后越改不动。方案一详情 →

方案二 · 止血 不改工具，用 helm + kubectl hack 出切换步骤

完全维持 milvus-helm/operator 现状，把切换 hack 成几步（以 pulsar → woodpecker 为例）：① 一次 helm upgrade --set woodpecker.enabled=true 装上 woodpecker service，并用 user.yaml 把旧 pulsar 钉死（盖过 chart 渲染，两套 MQ 并存）；② 调 milvus 管理 API wal/alter 在线切到 woodpecker；③ 切换确认后，用 k8s 原生命令删掉旧 pulsar 资源。当天可用，但顺序敏感、无护栏。方案二详情 →

方案三 · 终态 无状态 WebUI，通过 k8s 操作各 service 的原生 operator & ConfigMap

把上面这些操作收进一个无状态 WebUI：它不自带状态，通过 k8s 操作 milvus / pulsar / kafka / wp 各 service 各自的原生 operator 与 ConfigMap，落地上面那张"各管各"的架构，并把方案二的人肉步骤变成有护栏的向导。就是你正在用的这个原型。方案三详情 →

Notes · 吐槽一个真实的痛：流程文档多到维护不过来 😮‍💨

环境差异 × 安装方式 × 组件组合 → 文档爆炸（这也是 webui 想顺带解决的痛点诉求）

现在光是「安装 / 切换 / 改配置」这些流程，就被 环境差异 × 安装方式 × 组件组合 炸成一大堆 —— 每种组合都得单独维护一篇文档，写的人累、用户看着也晕、还容易过时。举个真实例子：更新一波文档后，想逐流程验证"步骤能不能一一对应、能不能跑通"，光罗列出来就近 30 个流程：
