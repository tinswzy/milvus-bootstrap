方案一 · 扩展 helm / operator

在不另起炉灶的前提下改造现有 helm/operator：把"安装"和"配置"彻底拆开 —— 安装工具不再假设依赖互斥，配置交回 ConfigMap 模板渲染。趁 3.0 一起改，越往后越改不动。过渡方案。

核心改动一句话：把 *.enabled（装+连共用一个开关）拆成两件独立的事 —— 安装 provision（装哪些组件，可多选、不假设互斥）+ 配置 active（Milvus 连哪个，由 configmap 模板渲染、单选）。

要改什么 · 直接对着源码

milvus-helm 解互斥 + 方便删旧 MQ；milvus-operator 待补两点

两件事拆开：安装（装哪些，各看各的 *.enabled，互不互斥）与 配置（连哪个，读一个显式开关）。下面是实际要动的位置。

① milvus-helm 解互斥 · charts/milvus/templates/config.tpl

# 现状：一条 else-if 优先级链，只有一个分支能渲染 mq.type —— 这就是互斥根源
{{- if .Values.externalPulsar.enabled }}   mq.type: pulsar
{{- else if .Values.pulsar.enabled }}     mq.type: pulsar
{{- else if .Values.woodpecker.enabled }} mq.type: woodpecker
{{- else if .Values.pulsarv3.enabled }}   mq.type: pulsar
{{- end }}

# 改成：连——只读一个显式开关；装——交给各自 *.enabled，不再 else-if 串成互斥
mq:
  type: {{ .Values.messageQueue.active }}   # 显式单选 pulsar|kafka|woodpecker

② milvus-helm 让 helm 能删旧 MQ

  • 子 chart（requirements.yaml）已带 condition: pulsar.enabled / kafka.enabled —— 关掉即 helm upgrade 自动删它，天然可删
  • 唯独 woodpecker 是手写 STS（templates/woodpecker-statefulset.yaml · -svc.yaml），要补 {{- if .Values.woodpecker.enabled }} … {{- end }} 包裹，才能被 helm upgrade 一并下线

③ milvus-operator · 待补两点（当前还没实现）

operator 的 ReconcileMsgStream（pkg/controllers/dependencies.go）现在只装 kafka / pulsar，woodpecker 走 default: do nothing。要跟上 helm，需补：
  • 支持安装 woodpecker：woodpecker 已是独立 service，不能再当"内置 MQ 不装"，要像 kafka/pulsar 一样真正把它装起来
  • 支持多个 built-in MQ 服务的安装与删除：切换窗口里能同时装新旧两个 MQ，切完后能干净删掉被换下的旧 MQ（不只是现在的单选 switch）

为什么能生效

装与连解耦后，"切换窗口"才表达得出来：先装好新 MQ（仍连旧的）→ 校验就绪 → 把 messageQueue.active 切到新 MQ（触发 milvus 重建）→ 再关掉旧 MQ 的 enabled。每步一次 helm upgrade。
删除靠现成机制：helm upgrade 会把 condition 变 false 的子 chart 删掉（PVC 默认保留）；woodpecker STS 加 if 包裹后同样能被 upgrade 一并下线 —— 不需要另造删除逻辑。
代价：越改越不纯粹。active × provision × standalone/cluster × internal/external 测试矩阵指数膨胀；我们仍在用自己维护的 pulsar/kafka 子 chart 代管别人的中间件。它买的是时间，不是终局 —— 见 方案三。
