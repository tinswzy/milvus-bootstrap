总体设计 · 架构 / 模块 / 流程

把前面讨论的能力模型（发现 → 识别 → 接管 → 安装 → 状态护栏）落成一套可落地的系统设计：整体架构、模块切分、核心流程、后台任务。先做 Python CLI 内核，WebUI 是后期薄壳。分工拆分后续单独做。

看 etcd 样板 →
一句话定位：一个无状态、有护栏的控制内核 —— 把「装 / 发现 / 接管 / 切换 / 改配 / 升级 / 删除」这些跨异构组件的运维动作，收敛成统一、幂等、可回滚、可审计的命令；每个组件仍走各自官方方式（helm / operator / external），工具不越界代管。

设计原则 · 六条

所有模块决策都回到这几条
🔀 装 / 连分离

「安装一个依赖」与「让 Milvus 连上它」是两件事。依赖可独立装、可被多个 Milvus 复用；连接才是渲染 config 的单选项。

🤝 各走官方方式

etcd→helm、minio→operator、kafka→strimzi… 工具编排它们各自的原生方式，不自己重写一套部署，不越界代管 Pod。

🛡️ 护栏前移

dry-run 预演、动权威数据先备份、危险操作二次确认、状态分级护栏 —— 把出事的可能挡在执行前。

📦 瘦 CLI + core daemon

core daemon 跑全部逻辑、命令行即可交付；CLI 是瘦客户端（装/启停/转发，还能自更新 core，像 claude code/codex）。WebUI 后期薄壳。

☁️ 尽量无状态

状态从集群派生，自身状态 piggyback 原生对象（labels/annotations + Secret/ConfigMap，仿 helm），MVP 不引入重 CRD。

♻️ 幂等可重入

每步先查现状再决定动作，重复执行安全；失败可回滚、可断点续跑、全程留审计。

服务知识库 · profile schema

这是扩展点 —— 加一个组件 ≈ 加一份 profile，六引擎自动复用
# profiles/etcd.yaml —— 驱动全部引擎的声明
kind: etcd
state_class: authoritative      # 权威有状态：删=丢数据；成员级可由 quorum 重建
detect:                          # ② Identification 的检测签名
  image_match: ["*etcd*"]
  ports: [2379, 2380]
  helm_chart: "etcd-*"
  exclude: { labels: { tier: control-plane } }   # 硬排除控制面 etcd
install_methods:                  # ④ Provisioner 的可选装法
  - id: bitnami-helm
    default: true                # ← 默认：与 milvus-helm/operator 同 chart 同镜像
    chart: bitnami/etcd
    values: { replicaCount: 3, persistence: { size: 10Gi } }
  - id: raw-statefulset
  - id: external            # 不装，只填 endpoints
health:                          # 判「就绪」
  workload_ready: "sts.readyReplicas == replicas"
  cluster_check: "etcdctl endpoint health --cluster"
scale_rules:                     # ⑤ Lifecycle 扩缩护栏
  replicas: odd-only
  step: one-at-a-time
  scale_down: "preStop etcdctl member remove"
  immutable_fields: [volumeClaimTemplates]   # 改 PVC→orphan 删+重建
backup: { type: etcd-snapshot, before: [delete, resize, adopt] }
connect:                         # 🔀 装/连分离：Milvus 怎么 external 连它
  milvus_values: "externalEtcd.endpoints"
为什么这么设计：六个引擎都不写死任何组件，全读 profile。minio / woodpecker / pulsar / kafka 各填一份（差异只在检测签名、默认装法、state-class、扩缩护栏、connect 方式），引擎零改动即复用 —— 这就是 etcd 样板能推广的根本。

数据与状态模型 · 工具自己的状态放哪

沿用我们讨论的结论：能从集群派生就不自己存；要存就 piggyback 原生对象
① 从集群派生（首选）

实例拓扑、健康、归属，绝大多数每次现查（Discovery+Identification）。真相永远是运行中的目标本身，工具不维护独立"权威态"。

② 管理元数据 · labels/annotations

挂在被管资源上：milvus-bootstrap/managed=true、install-method、owner、adopted-at —— 这就是「我们管过它」的印记。

③ 实例快照 · 可重建缓存

存一份 last-applied 作 update 快速 diff 缓存；可删可重建（删了重 discover）、后台定期校正。落点跟随平台：k8s→Secret、docker/单机→本地文件。

④ 任务 & 审计记录

每个后台任务一条记录（步骤/状态/dry-run 计划），+ 操作审计流。可落 ConfigMap/Secret 或外部库。

逻辑模型：Instance(一个 Milvus) ── 绑定 ──> 多个 Dependency(etcd/minio/mq…)，每个依赖带「归属态 + state-class + 安装方式」三个标签。状态落点跟随运行平台（k8s→Secret、docker/单机→本地），且都是可删可重建、定期自纠的缓存——不是权威态。不引入 CRD（MVP）：原生对象就够、零集群侧安装、RBAC 现成；将来要声明式编排再考虑 Instance CRD。

核心流程设计

每个流程都是 Task 引擎里的一条可恢复状态机
F1 安装新实例

装/连分离的主线：先把要新建的依赖各自装起来，再装 Milvus 以 external 连上它们。强调依赖顺序和就绪门。

并行装 etcd + 对象存储；各自就绪门通过（含集群级体检）
→
3
装 MQ

→
4
装 Milvus
以 external 连上三者（填 endpoints），等就绪
→
5
登记
写管理元数据 + 快照，标 Managed；失败按 ④③②① 回滚
F2 发现与接管

把存量（milvus-helm / operator / 裸装）的实例纳入统一管理 —— 即 etcd 样板的三层走一遍。

扫出疑似工作负载 + external 端点
→

先 snapshot → 写元数据 / 接管 helm release / 旁挂快照
护栏：控制面组件永不接管；operator 仍在管的，要么连其 CR 一起接管、要么先只读；接管前对权威态组件强制备份。
F3 ★ 切换 MQ（旗舰流程）

装/连分离的最佳体现：脏活是「装新 MQ」，切换本身是干净的内核级 wal/alter。全程护栏 + 可恢复。

直面 helm diff 机制与 immutable 字段（前面讨论的结论在这里落地）。

F5 · 改配置

读 ConfigMap → diff 预览 → 应用 → 按需滚动重启 → 校验。权威态变更前快照。

F6 · 扩缩容 / 删除

state-class 驱动护栏：etcd 奇数/一次一个/member remove；minio 加 pool；wp 先 decommission；权威态删除强制备份 + 输名确认；PVC 默认保留并告警。

后台任务设计 · Orchestration 引擎

上面每个流程都跑在这套引擎上 —— CLI 同步跑、API 异步跑，同一份代码

运维动作大多多步、长耗时、易中断，必须做成可恢复的工作流。一个 Task = 一串 Step，每个 Step 四件套：

precheck
幂等 · 已完成则跳过
→
do
执行实际动作
→
verify
校验结果 / 就绪门
↘
compensate
失败时逆序回滚
任务状态机（持久化 → 可断点续跑）
Pending→ Running · step k→ Succeeded /Failed→RolledBack /Paused（可恢复）
幂等 / 可重入

每个 precheck 先经 Discovery/Identification 现查状态再决定做不做 —— 重复跑、续跑都安全，和「无状态、从集群派生」一脉相承。

持久化 / 可恢复

任务记录落原生对象（Secret/ConfigMap）：CLI 跑挂了重开能续，API 重启不丢任务。

dry-run 预演

每步支持「只出计划不执行」→ 就是 WebUI 安装/切换向导里给用户看的预览。

回滚 / 补偿

失败时逆序跑 compensate（如切 MQ 在切流前失败 → 清掉新装的 MQ、保留旧的不动）。

并发锁

同一 Instance 上加 lease/annotation 锁，避免两个操作同时改一套 Milvus。

进度事件

结构化事件流：CLI 打印进度条，API 经 WebSocket 推到前端实时回显。

所有逻辑都在 core daemon —— Task 引擎跑在 core 里。CLI 和 WebUI 都是瘦客户端，经同一套 Core API 提交任务、订阅进度；CLI 额外负责装 / 启停 / 更新 core 本身（自更新，像 claude code / codex）。这正是「先做 CLI、WebUI 薄壳」成立的前提。
