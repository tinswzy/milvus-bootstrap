方案二 · K8s 原生操作

完全不改 helm/operator，用 helm upgrade + kubectl 把切换 hack 成几步（以 pulsar → woodpecker 为例）。当天可用的止血手册。

把切换 hack 成三步

pulsar → woodpecker，全程不改 helm/operator

① 一条 helm upgrade：装上 woodpecker service + 用 user.yaml 把旧 pulsar 钉死

# 1a) 写一个 user.yaml，把 milvus 的 MQ 强行钉在旧 pulsar（user.yaml 优先级最高，盖过 chart 渲染）
$ cat >/tmp/p-to-w-recover-pulsar.yaml <<'EOF'
extraConfigFiles:
  user.yaml: |-
    log:
      level: debug
    mq:
      type: pulsar
    messageQueue: pulsar
    pulsar:
      address: p-to-w-pulsarv3-proxy
      port: 6650
EOF

# 1b) 一条 upgrade：装上独立 woodpecker service，并应用上面的 user.yaml
$ helm upgrade -i p-to-w charts/milvus \
    -f charts/milvus/values.yaml \
    -f /tmp/p-to-w-recover-pulsar.yaml \
    --reuse-values \
    --set woodpecker.enabled=true \
    --set streaming.enabled=true \
    --set streaming.woodpecker.embedded=false \      # woodpecker 跑成独立 service
    --set woodpecker.image.repository=harbor.milvus.io/woodpecker/woodpecker \
    --set woodpecker.image.tag=dev-204302e-82-all \
    --set image.all.repository=harbor.milvus.io/milvusdb/milvus \
    --set image.all.tag=master-20260609-1853a148 \
    --set woodpecker.containerSecurityContext.runAsUser=0 \   ## 临时设置为 root user
    --set woodpecker.containerSecurityContext.runAsGroup=0
一波同时干两件事：① 装上独立的 woodpecker service（woodpecker.enabled=true + streaming.woodpecker.embedded=false）；② 用 extraConfigFiles / user.yaml 把 milvus 的 mq.type 强行钉死在旧 pulsar（user.yaml 优先级最高，直接盖过 chart 的 MQ 渲染）。升级后：woodpecker 已就位、milvus 仍稳连 pulsar，两者并存。镜像 tag、临时 root 都得手填 —— 够 hack。

② 切换 WAL：调 milvus 的 streaming 管理 API（确认集群已抽干 / 可清空后）

# 运行时把 WAL 后端从 pulsar 切到 woodpecker —— 不改 helm、不重启、不碰 if-else 链
$ curl -X POST http://<milvus-proxy>:9091/management/wal/alter \
    -d '{"target_wal_name":"woodpecker"}'
真正的切换在内核运行时：3.0 streaming 架构把 WAL 选择做成了可在线 alter 的能力，一个管理 API 就把后端从 pulsar 换成 woodpecker —— 不依赖 helm/operator 重渲染，绕开了那条 MQ if-else 优先级链。这是切换能"干净"的关键。

③ k8s 原生删除旧 pulsar 实例资源

# 切换验证通过后，旧 pulsar helm 不会自动清，直接用 kubectl 删
$ kubectl delete sts,deploy,svc,cm -l app.kubernetes.io/instance=p-to-w-pulsarv3
$ kubectl delete pvc -l app.kubernetes.io/instance=p-to-w-pulsarv3   # 最后一步，不可逆

为什么这套 hack 能成

user.yaml 盖过 chart 渲染：milvus 读 user.yaml 的优先级高于 chart 渲染出的配置，所以不管模板里 MQ if-else 怎么算，都被 user.yaml 里的 mq.type: pulsar 强行钉住 —— 装新服务时连接不被改乱。
切换在内核运行时完成：3.0 streaming 把 WAL 选择做成可在线 alter 的能力，POST /management/wal/alter 直接把后端从 pulsar 换成 woodpecker，不重启、不重渲染、绕开 if-else 链 —— 这是切换能"干净"的关键。
有回滚窗口：删 pulsar PVC 之前都可退 —— API 把 target_wal_name 切回 pulsar 即可（user.yaml 里的 pulsar 配置还在），旧 PVC 在、数据可用。第三步删 PVC 不可逆，务必最后做。
为什么体验差：手搓 user.yaml 覆盖配置、手填镜像 tag、临时把 woodpecker 跑成 root、再调 milvus 内部管理 API 切换、最后人肉删 PVC —— 命令多、顺序敏感、无护栏，最致命的前提（数据不可迁移、PVC 不可逆）全靠人记得。这正是 方案三的切换向导 要把每一步、每个前提校验都变成有护栏的界面。
