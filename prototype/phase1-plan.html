B · 逐组件接入 · 统一模板（每个组件都走这一遍）

这就是循环体 —— 换组件只换内容、不换步骤
调研官方安装方式

确认这个组件业界有哪些装法（operator / helm / compose / external），各自怎么管 pod 资源、state-class 是什么。（详见部署调研存档）

本次选型

定这次接入哪种安装方式 + 理由（给默认）。MVP 平台是 k8s。

写 profile

detect 检测签名 · install_methods[平台] · state_class · health · scale_rules · connect（Milvus 怎么 external 连它）。

实现 ServiceDriver

继承 BaseServiceDriver，只 override 特殊语义（健康体检 / 扩缩护栏 / 备份 / 组件特有动作）。

接入引擎各层

让它在 Discovery → Identification → Ownership(接管) → Provisioner(install) → Lifecycle(升级/扩缩/配置/删) → State 各层按 kind 生效。

平台落地

走 K8sAdapter 路径（MVP）；DockerAdapter 路径排期到 docker 支持时。

端到端验证

install → health → connect 跑通；dry-run + 回滚可用；纳入回归。

C · 逐组件任务清单

按 MVP 顺序：etcd 先穿线，最后 milvus 把它们连起来
🗄️etcd选型：Bitnami HelmauthoritativeM1 · 第一个穿线
调研结论：官方无 operator（CoreOS etcd-operator 已归档）→ 主流是 Bitnami Helm（StatefulSet + headless + PVC，静态成员表凑 Raft 多数派）。硬护栏：识别时排除 kube-system 控制面 etcd。
本次选型：bitnami-helm —— 与 milvus-helm/operator 今天同 chart 同镜像，纳管零摩擦。
  • profile：detect(image/2379/helm 标记, 排除 control-plane) · install=bitnami-helm · authoritative · connect=externalEtcd.endpoints
  • driver override：health=endpoint health 多数派 · scale=奇数/一次一个/member remove · backup=snapshot
  • 接入各层：Discovery 签名 · install · 升级(immutable→orphan 重建) · State
  • 验证：装 3 副本 → quorum 健康 → milvus 能 external 连
🪣MinIO选型：MinIO Operator（Tenant）authoritativeM1
调研结论：官方推荐 MinIO Operator + Tenant CRD（每 pool 一个 STS、operator 持有、纠删码对等）；milvus 今天用的是社区 helm subchart（distributed）。
本次选型：minio-operator（贴合"各走官方方式"）；helm-subchart 留作回退。
  • profile：detect(Tenant CR / STS / :9000) · install=minio-operator(apply Tenant) · authoritative · connect=externalS3.*
  • driver override：health=/health/cluster 写 quorum · scale=加 pool（不可原地缩）· backup=mc mirror
  • 接入各层：先确保 operator 就位 · install Tenant · 等 .status Initialized/green
  • 验证：装 tenant → green → milvus 连 external S3 → 建 bucket
🪶Woodpecker选型：Woodpecker Operatorcache-backedM1 · MVP 的 MQ
调研结论：woodpecker 仓库自带 operator（woodpecker.zilliz.io/v1alpha1 · WoodpeckerCluster）—— apply 一个 CR + configRef ConfigMap，operator 自动建 STS + 3 services + PDB，并自动处理 scale-down decommission、滚动升级。gossip 组网、元数据进 etcd、日志落对象存储 → pod 近无状态、PVC 仅缓存。milvus-helm 的手写 STS 作回退。
本次选型：woodpecker-operator —— 官方既然给了 operator，就按「各走官方方式」用它；还白拿 decommission/rolling/PDB（手写 STS 这些得自己补）。注意 operator 不管 etcd/对象存储，需把它俩 endpoints 渲染进 configRef ConfigMap。
  • profile：detect(WoodpeckerCluster CR / STS / gossip SEEDS / :18080) · install=woodpecker-operator · cache-backed · connect=streaming.woodpecker
  • driver override：health=/healthz · scale-down 由 operator 自动 decommission · 依赖：etcd+对象存储先就位
  • 接入各层：确保 operator 就位 · 用已装 etcd/minio endpoints 渲染 ConfigMap + 下发 WoodpeckerCluster CR · State(缓存态)
  • 验证：etcd+minio 就绪 → apply CR → 等 .status 就绪 → milvus 用 woodpecker 端到端
Ⓜ️Milvus（连接方）选型：operator / helm + external连接目标M1 · 把三者连起来
调研结论：milvus 自身经 milvus-operator(Milvus CR) 或 helm 安装；关键是以 external 方式连上前面装好的 etcd/对象存储/MQ（装/连分离的"连")。
本次选型：MVP 用 helm + external*（最直接，全控外部接线）；operator 路径并行支持。
  • profile：detect(Milvus CR/STS/:19530) · install=helm/operator · connect 汇总三依赖 endpoints
  • driver override：switch-mq（POST :9091 /management/wal/alter）—— milvus 特有动作
  • 接入各层：install(注入 external etcd/s3/mq) · 安装流程 F1 收口
  • 验证：etcd+minio+wp+milvus 一套端到端跑通（M1 达成）
📡🌊Pulsar / KafkaApache Helm / StrimziauthoritativeM2 · 扩展 MQ
调研结论：Pulsar=Apache Helm（STS/:6650）；Kafka=Strimzi operator（Kafka CR/:9092）。都是重 broker 集群，故排在 woodpecker 之后。
本次选型：Pulsar apache-helm、Kafka strimzi。各自一条同模板循环。
  • profile ×2：detect / install_methods / authoritative / connect=externalPulsar · externalKafka
  • driver override ×2：各自 health / scale 护栏
  • 接入各层 + 验证：milvus 可任选三种 MQ 之一连接
  • 打通后：switch-mq 三步（装新 MQ → wal/alter → 下线旧）即自然成立

D · 阶段一完成判据

循环跑到这些都绿，阶段一即完成
M1
MVP 穿线：etcd + minio + woodpecker 可独立装，milvus 以 external 连通三者，端到端跑通一套。
M2
扩展 MQ：pulsar / kafka 接入，milvus 可任选其一连接。
M3
运维 + switch-mq：各组件升级 / 配置可用；切换 MQ 三步护栏打通。
M4
纳管存量：接管已有 milvus-helm / operator 装的实例（含 docker 平台接入排期）。
说明：每个组件的循环都是「调研 → 选型 → profile → driver → 接各层 → 验证」同一套——这正是分层抽象的收益：骨架只搭一次，组件像填空一样接入。M5（WebUI）属于阶段二，给已做好的命令套界面。
