阶段一 · 实施计划

把「调研」和「设计」合并成一条可执行的实施线：先搭好 CLI + core 架构骨架，再用统一的接入模板逐个把组件接进来。每个组件都从「调研官方安装方式 → 本次选型 → 接入各层 → 验证」这个循环走，循环跑完即阶段一完成。

架构依据 →
架构依据见 总体设计 · 运行时 · 分层抽象；各组件的详细部署调研（pod 怎么管）见 部署调研存档。本页只讲怎么一步步把它实现出来。
优先级调整：先把不依赖 woodpecker 的链路闭环 —— milvus(operator) external 连 kafka/pulsar、kafka↔pulsar switch-mq、kafka/pulsar 与 milvus 各自独立升级。woodpecker service 排到最后：milvus-operator 对 external woodpecker service 的支持由外部同事负责、不在本项目范围，待其就绪后再启动对接。
验证原则（每个事项都自带"绿灯"）：每一项的完成判据 = 一条可跑的检查，分三档由轻到重 —— 🧪 单测（pytest，无需集群）→ 📋 dry-run（mb … --dry-run 看渲染对不对）→ ☸️ 实测（kubectl/mb 在真集群断言）。每步独立可验、独立提交，循序渐进、不攒大包。验证节奏：我先执行一遍 → 绿灯 → 清理/复位 → 你照步骤复跑一遍确认顺畅（每步页都带「🧹 清理/复位」）。骨架 A 由 pytest 套件覆盖（61 项通过）；逐步检查见下方 E · 逐步验收清单。

B · 逐组件接入 · 统一模板（每个组件都走这一遍）

这就是循环体 —— 换组件只换内容、不换步骤
调研官方安装方式

确认这个组件业界有哪些装法（operator / helm / compose / external），各自怎么管 pod 资源、state-class 是什么。（详见部署调研存档）

本次选型

定这次接入哪种安装方式 + 理由（给默认）。MVP 平台是 k8s。

写 profile

detect 检测签名 · install_methods[平台] · state_class · health · scale_rules · connect（Milvus 怎么 external 连它）。

实现 ServiceDriver

继承 BaseServiceDriver，只 override 特殊语义（健康体检 / 扩缩护栏 / 备份 / 组件特有动作）。

接入引擎各层

让它在 Discovery → Identification → Ownership(接管) → Provisioner(install) → Lifecycle(升级/扩缩/配置/删) → State 各层按 kind 生效。

平台落地

走 K8sAdapter 路径（MVP）；DockerAdapter 路径排期到 docker 支持时。

端到端验证

install → health → connect 跑通；dry-run + 回滚可用；纳入回归。

C · 逐组件任务清单

新顺序：etcd → minio → kafka/pulsar → milvus(连) → switch/升级；woodpecker 最后（待外部依赖）
🗄️etcd选型：Bitnami Helmauthoritative✅ 已实现
调研结论：官方无 operator（CoreOS etcd-operator 已归档）→ 主流是 Bitnami Helm（StatefulSet + headless + PVC，静态成员表凑 Raft 多数派）。硬护栏：识别时排除 kube-system 控制面 etcd。
本次选型：bitnami-helm —— 与 milvus-helm/operator 今天同 chart 同镜像，纳管零摩擦。
  • profile：detect(image/2379/helm 标记, 排除 control-plane) · install=bitnami-helm · authoritative · connect=externalEtcd.endpoints
  • driver override：health=endpoint health 多数派 · scale=奇数/一次一个/member remove · backup=snapshot
  • 接入各层：Discovery 签名 · install · 升级(immutable→orphan 重建) · State
  • 验证：装 3 副本 → quorum 健康 → milvus 能 external 连
🪣MinIO选型：MinIO Operator（Tenant）authoritative✅ 已实现
调研结论：官方推荐 MinIO Operator + Tenant CRD（每 pool 一个 STS、operator 持有、纠删码对等）；milvus 今天用的是社区 helm subchart（distributed）。
本次选型：minio-operator（贴合"各走官方方式"）；helm-subchart 留作回退。
  • profile：detect(Tenant CR / STS / :9000) · install=minio-operator(apply Tenant) · authoritative · connect=externalS3.*
  • driver override：health=/health/cluster 写 quorum · scale=加 pool（不可原地缩）· backup=mc mirror
  • 接入各层：先确保 operator 就位 · install Tenant · 等 .status Initialized/green
  • 验证：装 tenant → green → milvus 连 external S3 → 建 bucket
📡🌊Kafka / PulsarKafka=Helm(默认)·Strimzi / Pulsar=Apache Helmauthoritative✅ 已实现① 先闭环的 MQ
调研结论：Kafka=Strimzi operator（Kafka CR/:9092）或 Bitnami Helm；Pulsar=Apache Helm（STS/:6650）。不依赖 woodpecker —— 先用它们把整条链路闭环。
本次选型：Pulsar apache-helm、Kafka bitnami-helm（最快闭环；Strimzi operator 留作后续官方化）。各自一条同模板循环，可独立升级。
  • profile ×2：detect / install_methods / authoritative / connect=externalKafka.brokerList · externalPulsar.endpoint
  • driver override ×2：各自 health / scale 护栏；独立升级（与 milvus 解耦）
  • 接入各层 + 验证：milvus 经 operator 以 external 连 kafka 或 pulsar
  • switch-mq（核心）：kafka ↔ pulsar 三步（装新 MQ → wal/alter → 下线旧）真集群跑通
Ⓜ️Milvus（连接方）选型：milvus-operator + external连接目标✅ 已实现② operator 连 kafka/pulsar
调研结论：milvus 经 milvus-operator(Milvus CR) 或 helm 安装；关键是以 external 方式连上前面装好的 etcd/对象存储/MQ（装/连分离的"连")。
本次选型：milvus-operator（Milvus CR，dependencies.*.external 注入外部 etcd/minio/kafka|pulsar，milvus 不装任何依赖 pod）；helm 路径作回退。
  • profile：detect(Milvus CR/STS/:19530) · install=operator-cr/helm · connect 汇总依赖 endpoints
  • driver override：switch-mq（POST :9091 /management/wal/alter）· 升级（镜像滚动）—— milvus 特有动作
  • 接入各层：install(CR 注入 external etcd/s3/mq) · 升级/扩缩/配置/删 · 安装流程收口
  • 验证：etcd+minio+(kafka|pulsar)+milvus 一套端到端跑通（M1 达成）
🪶Woodpecker（service）选型：Woodpecker Operatorcache-backed⏸ 暂缓 · 待外部依赖
为何排最后：milvus-operator 对 external woodpecker service 的支持由外部同事负责、不在本项目范围；在它就绪前这条线无法闭环，故放到最后、等依赖落地再启动对接。（早前为打通临时改过 operator feat/external-woodpecker 分支，仅作交接参考，不并入本项目主线。）
本次选型：woodpecker-operator（woodpecker.zilliz.io/v1alpha1 · WoodpeckerCluster）—— 白拿 decommission/rolling/PDB；operator 不管 etcd/对象存储，需把二者 endpoints 渲染进 configRef ConfigMap，且 service 模式服务端需自带 quorum seeds。
  • 前置（外部）：milvus-operator 支持 dependencies.woodpecker.external(service) 就绪
  • profile：detect(WoodpeckerCluster CR / gossip SEEDS / :18080) · install=woodpecker-operator · cache-backed · connect=woodpecker.external
  • 接入各层：用已装 etcd/minio endpoints 渲染 ConfigMap + 下发 WoodpeckerCluster CR · State(缓存态)
  • 验证：依赖就绪 → apply CR → service 3/3 → milvus(≥3.0) 用 woodpecker service 端到端

D · 阶段一完成判据

循环跑到这些都绿，阶段一即完成
M1
MVP 穿线（不依赖 woodpecker）：etcd + minio + (kafka 或 pulsar) 可独立装，milvus 经 operator 以 external 连通，端到端跑通一套。
M2
双 MQ + 切换：kafka 与 pulsar 都接入，milvus 可任选；kafka ↔ pulsar switch-mq 三步护栏真集群打通。
M3
运维闭环：kafka/pulsar 各自独立升级、milvus 升级、配置 get/set/restart 可用。
M4
纳管存量：接管已有 milvus-helm / operator 装的实例（含 docker 平台接入排期）。
M5
woodpecker service（最后 · 待外部依赖）：外部同事把 milvus-operator 的 external woodpecker service 支持做完后，接入 woodpecker-operator service 模式并端到端验证。
说明：每个组件的循环都是「调研 → 选型 → profile → driver → 接各层 → 验证」同一套——骨架只搭一次，组件像填空一样接入。本次调整：M1–M3 先做（kafka/pulsar 闭环 + switch + 升级），M5 woodpecker 放最后、由外部依赖驱动。WebUI 属于阶段二，给已做好的命令套界面。
