🧭 关键决策

  1. 命令形态：单命令 mb doctor 全包（环境 / 版本 / 兼容 / 工具四段）。
  2. 镜像检查：manifest 轻量探测（registry 可达）+ 节点缓存检查，不真拉。
  3. 兼容权威：数据驱动可编辑；milvus↔MQ/woodpecker = 有把握的硬约束；operator/helm/k8s↔milvus 由用户权威表填入，未填=只提示不硬拦。
  4. 工具更新：显示版本 + git commit，能访问 remote 时比对最新 tag/commit，访不到=不可用。
  5. 门禁：hard 冲突→拦截 + --force 逃生口；soft/未知→只 WARN；switch-mq 同类切换=硬拦。
  6. 输出：rich 表 + --json，本地优先、优雅降级。

🏗️ 架构 · 模块划分

兼容判定单一真相，doctor 与门禁共用
模块职责
core/compat.py（扩展）兼容判定唯一真相：加载 YAML 矩阵 + 保留现有 MQ 硬规则；暴露 evaluate(versions) 与 gate(op, ctx)
core/compat.yaml（新·可编辑）数据驱动版本约束表（component × milvus 版本区间 × min/max × severity × source）
core/probe.py（新）只读版本探测：k8s / operator / helm / minio / kafka/pulsar / milvus 实例 / woodpecker，best-effort，探不到→None
core/doctor.py（新）编排：环境检查 + probe + compat.evaluate + 工具版本 → DoctorReport
cli/main.py（改）新增 doctor 命令（rich / --json）；给 install/upgrade/switch-mq 加 --force 接线门禁
边界：doctor 直接只读探测、不经 daemon；daemon 自身作为"环境"里的一个被检查项 → 预检在 daemon 未起时仍可用。

🧩 兼容矩阵数据模型（compat.yaml）

core/compat.yaml
constraints:
- component: milvus-operator # 被约束的组件
requires: milvus
milvus_range: ">=2.6.0,<3.0.0"
min: "" # ← 待权威表填；留空=未知→只提示不硬拦
severity: soft # hard=硬拦 / soft=只WARN
source: user-table # confident / best-effort / user-table
milvus↔kafka/pulsar/woodpecker 的硬规则（含 2.6 embed / 3.0 service、kafka/pulsar 下限）仍由 compat.py 的 MQ_OPTIONS 承载，YAML 不重复。operator/helm/k8s 建好槽位待你的权威表。

⛔ 门禁示例

硬拦 + --force 逃生口
mb install milvus --set image=milvusdb/milvus:v2.5.20 --set mq=woodpecker-service
✗ FAIL woodpecker-service 需 milvus ≥ 3.0.0（当前 2.5.20）→ 已拦截
# --force 打印警告后放行（自担风险）
mb switch-mq milvus-dev --to kafka # 当前已是 kafka
✗ 目标 MQ 与当前同类，switch-mq 不允许同类切换

☑️ 验收（DoD）

  • mb doctor 真集群跑出四段表；--json 可解析；有 FAIL 时退出码非 0
  • 至少一条真实约束（milvus 2.5 + woodpecker-service）被门禁硬拦、--force 可放行
  • switch-mq 同类切换被拒
  • 纯单测全过（矩阵加载 / evaluate / gate / 版本解析 / doctor 分段 + 退出码）
