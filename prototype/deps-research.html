阶段一调研 · 依赖部署与 Pod 管理

阶段一 MVP 要独立装起 etcd / MinIO / Woodpecker 三件套，再让 Milvus 以 external 直连它们。这页把三者各自的官方部署方式、以及 k8s 里怎么管理对应的 Pod 资源讲清楚 —— 作为 milvus-bootstrap CLI 的实现依据。

← 回实施计划
三件套都走同一个 k8s 套路：StatefulSet + headless Service 给每个 Pod 稳定的序号身份和域名。但把这些 Pod「编排成一个集群」的方式三者完全不同 —— etcd 用静态成员表凑 Raft 多数派、MinIO 用纠删码对等分片、Woodpecker 用 gossip 组网 + 状态外置到 etcd 和对象存储。看懂这三种模型，才能写对 CLI 的「装 / 判就绪 / 扩缩容 / 卸载」。

怎么管理 Pod 资源

  • 谁管 · StatefulSet 控制器：Pod 固定序号身份 etcd-0/1/2，配合 headless svc 拿到稳定 DNS。
  • 组网 · 模板把每个序号的 peer URL 拼成静态 ETCD_INITIAL_CLUSTER；podManagementPolicy: Parallel 让 3 个 Pod 同时启动凑齐 Raft 多数派（顺序启动会死锁）。
  • 容错 · 写入要多数派 (n/2)+1，所以副本取奇数 —— 3 容 1、5 容 2；偶数只增成本不增容错。
  • 重启/缩容 · Pod 重建后挂回同一块 PVC、保持成员身份；缩容靠 preStop 钩子 etcdctl member remove 注销，硬删会留下僵尸成员要手动清。
依据：milvus-helm charts/milvus/values.yaml:688-738；etcd-6.3.3 chart templates/statefulset.yaml（initial-cluster 循环 / Parallel / preStop / volumeClaimTemplates）。

怎么管理 Pod 资源

  • 谁管 · StatefulSet 控制器；走官方 operator 时是 每个 pool 一个 StatefulSet、由 operator 持有（非自家创建的 STS 会被判 StatusNotOwned）。
  • 组网 · 无主、节点对等：靠有序「省略号」端点展开 minio-{0...N-1}.headless 让每个节点知道全量成员，没有 leader / 协调者 / name-node。
  • 数据 · 一个对象按纠删码切成 N=数据+校验 片，一片一盘散在一个 erasure set 里；读要 quorum、写要 quorum，可容忍若干块盘/节点掉线。
  • 扩缩容 · 分布式 MinIO 不能原地缩、也不能原地扩单 pool —— 扩容是加一个新 pool（operator 里往 spec.pools[] 追加，新起一个 STS）；要下线得先 mc admin decommission 把数据迁走。
依据：milvus-helm charts/milvus/values.yaml:627-682 + minio subchart templates/statefulset.yaml（端点展开 / volumeClaimTemplates）；官方 minio/operator Tenant CRD 与 distributed DESIGN 文档。

怎么管理 Pod 资源

  • 谁管 · StatefulSet 控制器（仓库里那套 WoodpeckerCluster operator 能自动管生命周期，但 MVP 不启用）。
  • 组网 · gossip / memberlist：模板把全部 Pod 域名拼进 SEEDS 环境变量，节点经 gossip 端口互相发现；客户端连任一节点即可拿到全拓扑。
  • 状态外置 · 元数据写 etcd、日志(WAL)落 对象存储，本地 PVC 只是缓存/暂存 → Pod 近似无状态、好扩好换。这正是它「最省」的原因：复用 Milvus 本就需要的 etcd + 对象存储，不再多起一套 broker 集群（不像 Pulsar/Kafka）。
  • 扩缩容 · 扩容直接加序号即自动入群；缩容前要先 decommission 把段 flush 到对象存储 —— helm 路径不会自动做，CLI 得自己调 admin API 再降副本。
依据：milvus-helm templates/woodpecker-statefulset.yaml（SEEDS 循环 / MINIO_* / STORAGE_TYPE=service / volumeClaimTemplates）、woodpecker-svc.yaml；woodpecker README.md 架构、config/woodpecker.yaml（meta.type=etcd / storage.type=minio）、deployments/operator/。

给 CLI 的启示 · 四件要落地的事

三种模型 → 四处实现差异
1判「就绪」：基线一样，集群级各不同

三者都先看 readyReplicas == replicas，再各加集群级体检 —— etcd：etcdctl endpoint health 看多数派；MinIO：/minio/health/cluster 看写 quorum；Woodpecker：/healthz + 成员就绪。注意 headless 的 publishNotReadyAddresses 会让域名先于就绪解析，别拿 DNS 能解析当就绪。

2安装顺序：地基先行

etcd & 对象存储是地基 → 再起 Woodpecker（启动脚本会 nc 等 MinIO，没起来会卡）→ 最后装 Milvus 以 external 直连三者。这个硬依赖顺序就是 CLI 编排状态机的主线。

3扩缩容：三套不同的护栏

etcd 必须奇数、一次一个，缩容靠 preStop member remove；MinIO 只能加 pool、不能原地缩；Woodpecker 缩容前必须先 decommission flush 到对象存储。CLI 要按组件分别加护栏，不能一把 scale 了事。

4卸载：PVC 不会自己走

StatefulSet 的 volumeClaimTemplates PVC 不会被 helm uninstall / 缩容删除。CLI 的 teardown 要显式 kubectl delete pvc -l ...，否则残留盘越积越多（也是误删数据的风险点，需二次确认）。

一句话结论：「各组件用各自官方方式」不是统一 operator —— 三件套里只有 MinIO 有成熟官方 operator（Tenant CRD），etcd 基本只有 helm，Woodpecker 自带 operator 但尚新（MVP 走 helm/STS）。CLI 内核要做的，正是把这三种异构安装方式 + 三套 Pod 管理护栏，收敛成统一的「装 / 判就绪 / 扩缩容 / 卸载」命令。
一处待确认（开发内部）：milvus-helm 的 Woodpecker StatefulSet 目前只往 Pod 注入了 MINIO_*，没有传 ETCD_ENDPOINTS、也没挂配置文件。standalone 节点的 etcd 元数据地址从哪来（镜像内默认值？还是 chart 漏配？）需要和 chart owner 对一下 —— 这直接影响 CLI 装 Woodpecker 时要不要自己补 etcd 接入。
