① 本机 core

core daemon 作为本地进程/容器跑在运维机上，CLI 连本地 socket；管本机 docker，或经 kubeconfig 管远程 k8s。最轻，MVP 主形态。

② 集群内 core

core 作为 Deployment + ServiceAccount + RBAC 跑在 k8s 里，暴露 REST/WS 给多人 / WebUI；CLI 远程连它。

③ 主机常驻 core

core 作为 systemd / 容器跑在运维主机，集中管没有 k8s 的 docker 环境或多个目标；CLI 连它。

关键（修正）：真正干活的是 core daemon —— 六引擎 / 驱动 / 适配器 / Task / 状态 / 后台线程全在它里面。CLI 是瘦客户端，只做三件事：装 / 启停 / 更新 core（自更新，像 claude code / codex）、转发命令、显示进度；连「一次性命令」也是 CLI 先确保 core 起着、再 RPC 给它执行。core 跑在哪由需求定，CLI 在哪都能连。

② 状态结构与落点 · StateStore 抽象

状态是可重建缓存；落点跟随运行平台、定期自纠
# 逻辑状态结构 —— 全是可重建缓存（删了重新 discover 即可）
# 落点跟随平台：k8s 跑的→k8s Secret；docker/单机 跑的→本地 SQLite/JSON
Instance:                       # 一套 Milvus + 它的依赖绑定
  id / name
  platform: k8s | docker | standalone
  location: { namespace | host }
  milvus: { version, mode, endpoints }
  ownership: Managed | Adoptable | ReadOnly | External
  deps: [ DependencyBinding ]
  snapshot_ref                  # 仅供 update 时快速 diff 的缓存（可删可重建）
DependencyBinding:
  kind: etcd | minio | woodpecker | pulsar | kafka
  install_method: bitnami-helm | minio-operator | compose | external …
  ref: { release | container | endpoints }
  ownership / state_class / version / endpoints
Task: { id, type, target, status, steps:[{name,status,plan,result}], audit }
#  ↓↓ 以下是缓存：可重建、不权威，丢了重新拉即可  ↓↓
ImageStatus: { kind, current, latest, upgradable }
Candidate:   { kind, evidence, identified, suggested_ownership, status: pending_confirm }
HealthSnap:  { instance, workload_ready, cluster_health, ts }
数据性质落点 = 跟随实例运行的平台
Instance / 绑定清单可重建缓存删了重 discoverk8s→Secret/ConfigMap（带 label）　·　docker/单机→SQLite/JSON
实例快照 snapshot可重建缓存仅供 update 快速 diff同上（跟随平台）
Task / 审计记录记录丢了不影响正确性同上
image / candidate / health缓存内存为主 + 可选持久
连接 / 凭据配置输入用户给的，非状态k8s→挂载 Secret/env　·　单机→~/.milvus-bootstrap/config.yaml
三个修正要点：① 落点 = 跟随运行平台，不是可选项 —— 实例在 docker 跑则状态在本地、在 k8s 跑则状态在 k8s Secret，天然一一对应（K8sStateStore / FileStateStore 同一接口）。② 状态是可删的派生缓存，删了下次 discover 自动重建，只为 update 时快速检查 / diff 用。③ 后台定期拿真实状态校对、自动纠偏。永远的真相是运行中的目标本身，StateStore 只是加速用的便捷副本。

③ 平台抽象 · 同时管 k8s 与单机 docker

引擎平台无关，靠 PlatformAdapter 屏蔽差异
六引擎 · 平台无关Discovery / Identification / Ownership / Provisioner / Lifecycle / State 只调统一原语
↓
PlatformAdapter 接口
apply / get / delete workload · exec · logs · wait_ready · discover_native · state_store
↓
☸️ K8sAdapter
  • workload = helm release / operator CR / 原生对象
  • discover = 扫 STS/Deploy/CR + label
  • exec = pod exec · logs = pod log
  • state = Secret/ConfigMap
🐳 DockerAdapter
  • workload = container / compose project
  • discover = list 容器 + image + label
  • exec = docker exec · logs = docker logs
  • state = 本地文件 / SQLite
🖧 StandaloneAdapter （后续）
  • workload = systemd unit / 进程
  • discover = ssh 探测 + 端口
  • exec/logs = ssh · journalctl
  • state = 本地文件
两维抽象 = 平台 × 组件。「在哪跑」交给 PlatformAdapter；「装哪个组件、怎么认」交给组件 profile，且 profile 的 install_methods / detect 按平台分别声明。于是 etcd 在 k8s 上是 bitnami-helm、在 docker 上是跑容器 —— 同一个 Provisioner 调不同 adapter 即可，引擎零改动。

⑤ 后台线程 · 常驻 daemon 才跑

区别于按需触发的 Task 引擎 —— 这些是周期性自巡的 worker
core daemon · 单事件循环 + worker 池（阻塞的 helm/kubectl/docker 调用丢线程池）
🔭 Discovery poller
每数分钟
🏷️ Image/version poller
每数小时
❤️ Health poller
每 30–60s
⚙️ Task runner（worker 池）
常驻消费队列
🔀 Drift detector
每数分钟
🧹 GC / retention
每日
线程 / loop读什么产出 / 写是否自动动手
🔭 Discovery poller各 target 的工作负载 / 容器 / 端点刷新候选清单 + 校正已知实例的 state 缓存；新发现的 stateful / deployment / 容器 → 置 pending_confirm只标记 · 待用户确认才接入
🏷️ Image/version poller镜像 registry · helm repo · GitHub releases比对运行版本 → 标「可升级」+ 版本差只标记 · 待用户确认才升
❤️ Health poller被管实例：就绪门 + 集群级体检（etcdctl / minio cluster / wp healthz）更新状态、亮灯、触发告警只读
⚙️ Task runner任务队列执行用户已提交的 Task 状态机、推进度、重试 / 续跑执行已授权任务
🔀 Drift detectorlive 状态 vs last-applied 快照标记漂移（被 kubectl edit 过的字段）只标记
🧹 GC / retention旧任务记录 / 缓存 / 快照历史按保留策略清理自动（安全操作）
护栏：巡检线程只「发现」不「动手」。Discovery 发现新资源 → 进待确认队列，用户点确认才接管；Image poller 发现新版本 → 只亮「可升级」，用户点了才走 F4 升级。真正改动集群的只有 Task runner，且跑的都是用户已授权、带 dry-run / 回滚的任务。这样"自动巡检"和"人来拍板"分得很清。
这些线程都跑在 core daemon 里 —— CLI 是瘦客户端，只转发命令、看进度，不跑任何业务逻辑（含这些巡检）；core 没在跑时 CLI 会先把它拉起来。周期巡检也是 state 缓存能"定期自纠"的来源（Discovery/Health poller 拿真实状态回写校正）。
合起来看：总体设计讲「系统由什么组成」，这页讲「它怎么活起来」—— 一个管理面进程，按需 CLI / 常驻 daemon 两形态，靠 PlatformAdapter 同时罩住 k8s 与 docker，状态经 StateStore 落原生对象或本地文件，常驻态再开一圈只发现不动手的巡检线程。下一步仍是 计划拆分，把这些落成工作项。
